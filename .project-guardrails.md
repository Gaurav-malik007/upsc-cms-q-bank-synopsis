# Project Guardrails & Architecture Rules

## 1. Pre-Modification Protocol
- **Context Check**: Before changing any file, identify all existing functions and exported components.
- **No Deletion Policy**: Do not delete or refactor existing logic unless I explicitly ask you to 'refactor' or 'replace'.
- **Dependency Mapping**: If you modify a shared component (e.g., a Button or Navbar), check all files that import it to ensure they don't break.

## 2. Coding Standards
- **State Management**: Keep existing state patterns (e.g., Zustand/Context) consistent. Do not introduce new libraries without asking.
- **UI Consistency**: Follow the existing Tailwind/CSS patterns. Do not overwrite global styles that affect other pages.

## 3. Verification Step
- After every change, perform a **'Self-Review'**.
- **Check**: 'Does the previous core functionality (Login, Navigation, Data Fetching) still work?'
- **Automation**: Always run `npm run build` or `npm run lint` (if available) automatically after finishing a task to verify no regressions were introduced.
- If a change causes a linting error or a build break, revert the change immediately and notify me.

## 4. Error Handling
- Always wrap new features in basic error boundaries so a failure in a 'new update' doesn't crash the entire 'previous app'.
